# mdo v0.6 Plan: Quiet Output, Reliable Workflow

## Goal

Strengthen mdo’s central experience:

> **Open Markdown as HTML.**

The CLI remains the fast, dependable engine. File-manager integration remains the elegant everyday interface. Setup and generated-page branding should recede into the background.

## Product principles

1. Fast CLI behavior comes first.
2. File-manager integration is a core feature, not an optional side project.
3. Setup should be simple, reversible, and nearly invisible.
4. Generated pages should emphasize the source document, not mdo.
5. Automatic and manual light/dark modes remain essential.
6. Performance and diagnostic information belongs in the terminal.
7. Do not build lifecycle-management complexity before users demonstrate a need for it.

---

## Release scope

### 1. Fix correctness before changing presentation

#### Browser launch failures

When `--open` successfully renders but cannot launch the browser:

* print the error to stderr;
* print the rendered output path so the user can open it manually;
* exit non-zero.

Acceptance criteria:

* `mdo --open file.md` exits `0` only when both rendering and browser launch succeed;
* browser-launch failure does not delete the rendered file;
* file-manager wrappers propagate failure where practical.

#### Correct file URLs

Replace the hand-built `<base href>` URL conversion with a proper filesystem-to-URL implementation.

Acceptance criteria:

* paths containing spaces, `#`, `%`, `?`, Unicode, and Windows drive letters work correctly;
* relative images and links work when `--open` renders into the temporary directory;
* tests cover representative Unix and Windows paths.

#### Reliable watch mode

Watch the source file’s parent directory and filter events for the target file rather than relying only on `Modify` events on the file itself.

Handle:

* direct writes;
* truncate-and-rewrite saves;
* temporary-file-plus-rename atomic saves;
* file replacement;
* duplicate event bursts.

Acceptance criteria:

* saving from common editors continues to render after repeated atomic replacements;
* unrelated changes in the same directory do not trigger rendering;
* debounce behavior remains predictable;
* a failed render does not terminate watch mode.

#### Run tests on Windows

Change Windows CI from compile-only checking to running the applicable test suite.

Add targeted tests for:

* registry command construction;
* paths containing spaces and metacharacters;
* wrapper argument forwarding;
* browser-launch error propagation;
* setup launcher behavior that can be tested without modifying the real registry.

### 2. Make generated documents quieter

Remove the visible footer text containing:

* “Generated by mdo”;
* mdo version;
* render date;
* render duration.

Retain:

```html
<meta name="generator" content="mdo 0.6.0">
```

This preserves provenance without branding the visible document.

#### Source freshness

Display the source file’s modification time using restrained wording:

> Source modified: July 14, 2026, 10:42 AM

Use an appropriate `<time datetime="…">` element.

Important distinction:

* call it **source modified**, not “document date” or “content date”;
* filesystem modification time may reflect copying or checkout activity;
* do not imply stronger semantics than the metadata supports.

For v0.6, use filesystem modification time only. Explicit frontmatter fields such as `date:` or `updated:` can be considered later without introducing a YAML parser now.

Acceptance criteria:

* full-document output includes source modification time;
* bare output does not;
* timestamps are represented in a machine-readable form;
* missing or unreadable timestamp metadata does not prevent rendering;
* output contains no visible mdo branding.

---

### 3. Keep light/dark mode intact

Preserve:

* automatic OS-theme detection;
* the manual light/dark toggle;
* persistence of the user’s manual choice.

Review only the implementation quality:

* ensure the toggle is keyboard accessible;
* give it an accessible label;
* avoid obscuring document content on small screens;
* verify it works in local `file://` pages;
* ensure custom CSS can restyle or hide it.

This is not a candidate for removal or deferral.

---

### 4. Move performance information behind `--verbose`

Add:

```text
mdo --verbose file.md
```

Normal output remains minimal.

Verbose information goes to stderr and reports the whole workflow, not merely Markdown parsing:

```text
Rendered file.md → file.html in 14 ms
  read:       1 ms
  markdown:   3 ms
  sanitize:   2 ms
  assemble:   1 ms
  write:      7 ms
```

For `--open`, optionally include browser-launch initiation separately:

```text
  launch:     4 ms
```

Do not write performance information into generated HTML.

Acceptance criteria:

* output HTML is identical with and without `--verbose`;
* diagnostics go to stderr;
* total time covers the complete render workflow;
* stage totals are internally consistent;
* normal successful conversion output remains brief.

Avoid adding a separate `--profile` command unless later diagnostics need materially more detail.

---

### 5. Simplify setup without weakening integration

Keep all existing supported integration capabilities:

* Windows Explorer **Open as HTML**;
* Linux XDG **Open as HTML**;
* reversible per-user installation;
* `mdo-setup` desktop entry points;
* optional default-handler flow where the platform permits it.

Do not implement the full state-aware integration manager described in ADR 0004 for this release.

Instead, implement only a cheap owned-state check:

* if mdo’s expected handler registration is present, say that integration is already installed;
* offer to leave it unchanged or reinstall it;
* do not attempt to infer every partial, stale, system-wide, or effective-default state;
* keep `--install-file-manager` idempotent;
* keep `--uninstall-file-manager` limited to resources owned by mdo.

Update ADR 0004 to **Deferred** and record the reason:

> Full integration-state management adds substantial platform complexity without current evidence that it solves a common user problem.

Acceptance criteria:

* first-time setup remains one clear decision;
* rerunning setup does not misleadingly present the user as completely new;
* no package-manager detection or full-uninstaller functionality is added;
* no new persistent background services or configuration databases are introduced.

---

### 6. Rewrite the README around the actual product

The primary README should become considerably shorter.

Recommended structure:

1. **Open Markdown as HTML**
2. One-paragraph explanation
3. Install
4. Three essential commands
5. File-manager integration
6. Security note
7. Links to detailed documentation

The essential command section should center on:

```text
mdo FILE.md
mdo --open FILE.md
mdo --watch FILE.md
```

Move these subjects into dedicated documentation:

* maintainer release procedures;
* detailed Windows registry behavior;
* detailed Linux XDG behavior;
* terminal-emulator fallback implementation;
* package-manager publishing;
* metrics implementation;
* advanced CSS site-building examples;
* setup architecture and ADR discussion.

Suggested files:

```text
docs/usage.md
docs/file-manager-integration.md
docs/custom-css.md
docs/maintaining.md
```

Acceptance criteria:

* a new user can understand and run mdo without scrolling through implementation details;
* file-manager integration remains prominent;
* detailed information remains available but does not dominate the landing page;
* generated site documentation and README do not duplicate large sections unnecessarily.

## Proposed pull-request sequence

### PR 1 — Correct open and watch behavior

* browser-launch exit status;
* proper file URLs;
* parent-directory watch handling;
* cross-platform regression tests.

### PR 2 — Quiet generated pages

* remove visible mdo branding;
* replace render metadata with source modification time;
* preserve generator metadata;
* retain and improve the theme toggle.

### PR 3 — Add terminal diagnostics

* introduce `--verbose`;
* measure complete workflow stages;
* send diagnostics to stderr;
* remove timing from document generation.

### PR 4 — Simplify setup scope

* add minimal owned-registration detection;
* keep installation idempotent;
* defer the full integration manager;
* update ADR 0004 accordingly.

### PR 5 — Reduce documentation weight

* rewrite the README;
* move platform and maintainer details into focused documents;
* update generated project-site pages;
* correct any release-history omissions, including the missing 0.3.0 changelog section.

---

## Explicit non-goals

Do not add in v0.6:

* a full GUI;
* an embedded web server;
* live browser reload;
* themes marketplace or template system;
* package-manager autodetection;
* a general mdo uninstaller;
* comprehensive partial-state recovery;
* frontmatter parsing;
* telemetry;
* more output formats;
* conversion-engine feature expansion unrelated to reading Markdown.

---

## Release definition of done

v0.6 is ready when:

* `mdo FILE.md` remains fast and unsurprising;
* `mdo --open FILE.md` either opens successfully or fails honestly;
* watch mode survives atomic editor saves;
* relative resources work from paths containing reserved characters;
* generated pages retain excellent light/dark readability;
* visible document metadata describes the source rather than advertising mdo;
* timing appears only when explicitly requested;
* setup remains easy without becoming an integration-management application;
* the README explains the product before explaining its machinery.
