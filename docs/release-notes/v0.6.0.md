# mdo v0.6.0

**Open Markdown as HTML** — v0.6 is a reliability and polish release. Nothing
in the core job changed: `mdo FILE.md` still renders a styled, standalone
HTML page next to your source in a few milliseconds. What changed is that the
edges that used to bite — `--open` lying about success, watch mode dying on a
save, awkward folder names breaking relative images — no longer do, and the
generated page now foregrounds *your* document instead of advertising the
tool.

This is still a young project. It runs daily on the author's Windows and Linux
machines; macOS binaries are built and released but the least exercised, and
the macOS Finder integration is still a manual Automator recipe. Feedback,
including "this shouldn't exist," is genuinely wanted.

## What's new

### `--open` fails honestly
Previously `mdo --open` could report success even when no browser actually
opened. Now, if the launch fails after a successful render, mdo exits non-zero
and prints both the error and the path to the rendered file (which is left in
place) on stderr. On Linux, an opener that spawns but exits non-zero — e.g.
`xdg-open` with no configured browser handler — is treated as the failure it
is rather than a silent success. `mdo-open` propagates the child's exit status.

### Watch mode survives atomic editor saves
Many editors (Vim, VS Code, and others) save by writing a temp file and
renaming it over the original, which replaces the file's identity and used to
silently kill a file-level watch. `--watch` now watches the parent directory
and filters to the target file, so those saves keep re-rendering. Save bursts
are debounced into a single render, and a failed render does not stop the loop.

### Relative resources work from awkward paths
`<base href>` URLs are now generated with a proper URL library, so source
directories containing spaces, `#`, `%`, `?`, or non-ASCII characters — and
Windows drive letters, UNC paths, and `\\?\` extended-length prefixes — are
correctly percent-encoded. Relative images and links resolve from directories
that previously produced a broken base tag. Paths that genuinely cannot form a
valid `file://` URL omit the tag rather than emit a broken one.

### `--verbose` timing, terminal-only
New `mdo --verbose FILE.md` prints per-stage render timings (read, markdown,
sanitize, assemble, write, total; plus browser-launch initiation with
`--open`) to **stderr only**. The generated HTML is byte-for-byte identical
with and without the flag — no timing or diagnostic data is ever baked into
the page.

### Quieter, source-focused pages
The generated document no longer carries a visible "Generated by mdo" footer,
version, render date, or render duration. Provenance is a single
`<meta name="generator">` tag in the head. A `Source modified: …` line (with a
machine-readable `<time datetime>`) describes *your* file. The light/dark
theme toggle gained a state-aware label/tooltip, live OS-theme tracking until
you choose manually, safe `localStorage` handling on `file://` pages, and can
be restyled or hidden from custom `--css` via `#theme-toggle`.

### Setup that knows its own state
Interactive setup now detects whether mdo's own file-manager integration is
already installed and offers a single honest choice — leave it (default) or
reinstall — instead of blindly re-running. Detection asks one question and
infers no partial, stale, or system-wide state; uninstall still touches only
mdo-owned resources. (Fuller integration-state management was considered and
deliberately deferred — see ADR 0004.)

### Documentation
The README is rewritten around the product, with platform, CSS, and
maintainer detail moved into focused pages under `docs/`. Windows CI now runs
the full test suite instead of only compiling.

Full details: [CHANGELOG.md](../../CHANGELOG.md).

## Cutting the release (maintainer runbook)

These are the exact steps for maphew. Everything above the tag is already
prepared on branch `v0.6/release-prep` (PR #26).

### 1. Confirm the restack and merge bottom-up

The branches must first be restacked so every PR head descends from its current
parent branch and the stack contains current `main`. For this release, preserve
`main`'s `AGENTS.md` while restacking PR #23. Confirm fresh Linux and Windows
CI is green on all five repaired heads before merging.

Use merge commits throughout. Do **not** squash this stack: squash-merging breaks
the ancestry the later PRs depend on and produces conflicts when they are
retargeted.

For each PR, retarget explicitly, mark it ready, wait for the checks triggered
by the base change, pin the reviewed head SHA, and merge that exact head:

```bash
# PR 2 → main
gh pr edit 22 --repo maphew/mdo --base main
gh pr ready 22 --repo maphew/mdo
gh pr checks 22 --repo maphew/mdo --watch --fail-fast
head_sha=$(gh pr view 22 --repo maphew/mdo --json headRefOid --jq .headRefOid)
gh pr merge 22 --repo maphew/mdo --merge --match-head-commit "$head_sha"

# PR 3 → main
gh pr edit 23 --repo maphew/mdo --base main
gh pr ready 23 --repo maphew/mdo
gh pr checks 23 --repo maphew/mdo --watch --fail-fast
head_sha=$(gh pr view 23 --repo maphew/mdo --json headRefOid --jq .headRefOid)
gh pr merge 23 --repo maphew/mdo --merge --match-head-commit "$head_sha"

# PR 4 → main
gh pr edit 24 --repo maphew/mdo --base main
gh pr ready 24 --repo maphew/mdo
gh pr checks 24 --repo maphew/mdo --watch --fail-fast
head_sha=$(gh pr view 24 --repo maphew/mdo --json headRefOid --jq .headRefOid)
gh pr merge 24 --repo maphew/mdo --merge --match-head-commit "$head_sha"

# PR 5 → main
gh pr edit 25 --repo maphew/mdo --base main

head_sha=$(gh pr view 25 --repo maphew/mdo --json headRefOid --jq .headRefOid)

# PR 6 / release prep → main
gh pr edit 26 --repo maphew/mdo --base main

head_sha=$(gh pr view 26 --repo maphew/mdo --json headRefOid --jq .headRefOid)

```

After each merge, confirm the merge commit's `main` CI succeeds before
retargeting the next PR. Keep the stack branches until the entire sequence is
complete.

### 2. Sync and sanity-check main

```bash
git switch main
git pull --ff-only
test -z "$(git status --porcelain)"
grep '^version' Cargo.toml          # must read: version = "0.6.0"
head -20 CHANGELOG.md               # 0.6.0 section and actual release date
cargo test --all-targets --all-features --locked
```

### 3. Tag and push (this triggers the release build)

```bash
release_sha=$(git rev-parse HEAD)
git tag -a v0.6.0 -m "Release v0.6.0" "$release_sha"
test "$(git rev-list -n 1 v0.6.0)" = "$release_sha"
git push origin refs/tags/v0.6.0
```

`.github/workflows/release.yml` builds the Linux, macOS, and Windows archives
and publishes them to a GitHub Release, generating the `SHA256SUMS` asset. You
can also trigger it manually via **Actions → Release → Run workflow** against
an existing tag.

### 4. Publish `mdo-cli 0.6.0` to crates.io

The documented `cargo install mdo-cli` channel currently stops at 0.3.0, so
publishing the crate is part of this release rather than an optional follow-up.
From the clean, tagged release commit, with crates.io credentials configured:

```bash
cargo publish --locked --dry-run
cargo publish --locked
cargo search mdo-cli --limit 1       # must report 0.6.0 after indexing
```

The GitHub Release workflow does not publish to crates.io.

### 5. (Optional) Create / edit the GitHub Release

The workflow publishes the release; to attach these notes explicitly:

```bash
awk '/^---$/{exit} {print}' docs/release-notes/v0.6.0.md \
  > /tmp/mdo-v0.6.0-release-notes.md
gh release edit v0.6.0 --repo maphew/mdo \
  --title "v0.6.0" \
  --notes-file /tmp/mdo-v0.6.0-release-notes.md
```

Or, if creating it by hand instead of via the workflow:

```bash
awk '/^---$/{exit} {print}' docs/release-notes/v0.6.0.md \
  > /tmp/mdo-v0.6.0-release-notes.md
gh release create v0.6.0 --repo maphew/mdo \
  --title "v0.6.0" \
  --notes-file /tmp/mdo-v0.6.0-release-notes.md \
  --verify-tag
```

### 6. Post-tag: refresh package manifests (do NOT fabricate hashes)

Only after the release assets and `SHA256SUMS` exist, update these to `0.6.0`
with the real URLs and hashes from that release — see `docs/maintaining.md`
and `packaging/README.md`:

- `packaging/homebrew/mdo.rb` — `version` and the per-OS asset URLs + `sha256`.
- `packaging/scoop/mdo.json` — `version`, URL, and hash.
- `packaging/winget/Maphew.Mdo/0.6.0/…` — new version directory
  (`Maphew.Mdo.installer.yaml`, `Maphew.Mdo.locale.en-US.yaml`,
  `Maphew.Mdo.yaml`) mirroring the existing `0.5.0/` set, with the real
  installer URL and `InstallerSha256`.
- `packaging/README.md` — bump the "current manifests target" line to `v0.6.0`.

These are intentionally left untouched in the release-prep PR because their
checksums do not exist until the tagged build runs.
